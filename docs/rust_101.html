Lesson 01 · The basics

Variables & values

Bindings are immutable by default — you opt into mutation with mut. let is a runtime binding; const is a compile-time value.

Rust
let n = 0;          // frozen
let mut m = 0;      // opt in to reassignment
m += 1;
const CAP: usize = 6000;

const n = 0;        // frozen
let m = 0;          // reassignable
m += 1;
// no deep compile-time const

n = 0               # nothing is frozen
m = 0
m += 1
CAP = 6000          # "const" by convention only

var n = 0;          // reassignable by default
int m = 0;
m += 1;
const int CAP = 6000;

Rust freezes by default and you add mut. TS makes you remember const; Python and C# freeze nothing unless you ask.

What does the compiler say?
Forget the mut, then bump the counter:
let idle_nudges = 0usize;
idle_nudges += 1;

refused. Plain let is frozen.

error[E0384]: cannot assign twice to immutable variable `idle_nudges` help: consider making this binding mutable: `mut idle_nudges`
Lesson 02 · The basics

Functions

Typed parameters, one return type. The body's last expression is the return value — return is only for leaving early.

Rust
fn stamp(text: &str) -> u64 {
    let h = hash(text);
    h                   // last expression = return
}

function stamp(text: string): number {
  const h = hash(text);
  return h;             // explicit return required
}

def stamp(text: str) -> int:
    h = hash(text)
    return h            # explicit; type hint optional

ulong Stamp(string text) {
    var h = Hash(text);
    return h;           // explicit return
}

Only Rust makes the last expression the return value. The others always spell out return.

What does the compiler say?
One stray semicolon after the last line:
fn stamp(text: &str) -> u64 {
    let h = hash(text);
    h;
}

refused. The semicolon discards the value, so the function returns (), not u64.

error[E0308]: mismatched types — expected `u64`, found `()` help: remove this semicolon to return this value
Lesson 03 · The basics

Control flow & expressions

if and match produce values, so there's no separate ternary — the same if works in statement and value position.

Rust
let note = if n > 1 {
    format!("{n} lines")
} else {
    String::new()
};

const note = n > 1
  ? `${n} lines`
  : "";               // ternary: a separate construct

note = f"{n} lines" if n > 1 else ""
# conditional expression

var note = n > 1
    ? $"{n} lines"
    : "";             // ternary

The others bolt on a ternary because their if is a statement. Rust's if already returns a value.

What does the compiler say?
Delete the else:
let note = if n > 1 {
    format!("{n} lines")
};

refused. With no else, the false case is (), which isn't a String.

error[E0317]: `if` may be missing an `else` clause — expected `String`, found `()`

Want an optional? Say so: (n > 1).then(|| format!("{n} lines")).

Lesson 04 · Shaping data

Structs

A fixed layout: these fields, nothing else. No optional-by-accident keys, no undefined. In agentj, every tool returns a ToolOutcome.

Rust
pub struct ToolOutcome {
    pub text: String,
    pub ok: bool,
}

interface ToolOutcome {
  text: string;
  ok: boolean;
}

@dataclass
class ToolOutcome:
    text: str
    ok: bool

record ToolOutcome(string Text, bool Ok);

Same shape; the difference is enforcement. A Rust struct can't be built with a missing or extra field.

Methods sit in an impl block. The receiver says how they touch the value: &self reads, &mut self mutates (Lessons 07–08).

What does the compiler say?
Add a method that flips a failed outcome:
impl ToolOutcome {
    fn force_ok(&self) {
        self.ok = true;
    }
}

refused. &self is a shared, read-only borrow. Use &mut self to mutate.

error[E0594]: cannot assign to `self.ok`, which is behind a `&` reference
Lesson 05 · Shaping data

Enums & match

A struct is AND (all fields). An enum is OR (one shape at a time). agentj's AgentEvent is the whole agent-to-UI protocol; match forces you to handle every variant.

Rust
enum Event { Message(String), Done }

match ev {
    Event::Message(t) => show(t),
    Event::Done => finish(),
}

type Event =
  | { kind: "message"; text: string }
  | { kind: "done" };

switch (ev.kind) {
  case "message": show(ev.text); break;
  case "done": finish(); break;
}

match ev:
    case Message(text):
        show(text)
    case Done():
        finish()

var r = ev switch {
    Message m => Show(m.Text),
    Done      => Finish(),
};

Rust and C# force exhaustiveness — miss a case and it won't compile. TS needs a never-check; Python's match has none.

What does the compiler say?
Your match handles Message and four others, but you add a new Error variant to the enum and forget to handle it. Result?

refused.

error[E0004]: non-exhaustive patterns: `Event::Error(_)` not covered

Adding a variant hands you a to-do list of every match that must now decide.

Lesson 06 · Shaping data

Option & Result

No null, no exceptions. Absence is Option, failure is Result — both ordinary enums, so the type tracks them. agentj's safe_resolve confines a path and can fail.

Rust
fn resolve(p: &str) -> Result<PathBuf, String> {
    if bad(p) { return Err("escape".into()); }
    Ok(canonical(p))
}

function resolve(p: string): string {
  if (bad(p)) throw new Error("escape");
  return canonical(p);   // failure = exception
}

def resolve(p: str) -> str:
    if bad(p):
        raise ValueError("escape")
    return canonical(p)    # failure = exception

string Resolve(string p) {
    if (Bad(p)) throw new Exception("escape");
    return Canonical(p);   // failure = exception
}

The others hide failure in an exception channel you can forget to catch. Rust puts it in the return type — you must handle Err to reach the value.

What does the compiler say?
The ? operator propagates errors. Use it in a function that returns a plain struct, not a Result:
fn read_file(&self, args: &Value) -> ToolOutcome {
    let abs = safe_resolve(&self.root, path)?;
}

refused. ? returns the failure from this function, so the return type must be able to hold one.

error[E0277]: the `?` operator can only be used in a function that returns `Result` or `Option`
Lesson 07 · The memory model

Ownership

The concept people mean when they call Rust hard — and it's three rules. Every value has one owner; when the owner leaves scope the value is freed; assignment moves ownership. No garbage collector. This one has no equivalent elsewhere.

Rust
let a = String::from("hi");
let b = a;            // MOVE — a is now dead
println!("{a}");      // compile error E0382

let a = "hi";
let b = a;            // both alias the same value
console.log(a);       // fine — GC frees it later

a = "hi"
b = a                 # both names → same object
print(a)              # fine — refcount/GC frees it

var a = "hi";
var b = a;            // both reference the string
Console.WriteLine(a); // fine — GC frees it later

GC languages let any number of names alias a value and free it whenever. Rust picks a single owner so it can free deterministically, with no collector.

What does the compiler say?
Move a String, then use the old name:
let summary = first_line(&result, 80);   // borrow — fine
let owned = result;                      // move
let both = format!("{result} / {summary}");

refused. Line 2 moved result; line 3 reads a dead value. Fix: borrow (&result) or result.clone().

error[E0382]: borrow of moved value: `result`
Lesson 08 · The memory model

Borrowing

References let you look without owning. The rule: any number of shared reads &T, XOR one exclusive writer &mut T — never both. It kills the mutate-while-iterating bug at compile time.

Rust
for m in items.iter() {
    items.push(x);    // compile error E0502
}

for (const m of items) {
  items.push(x);      // compiles; loops oddly / forever
}

for m in items:
    items.append(x)   # runs; can loop forever

foreach (var m in items)
    items.Add(x);     // throws at runtime:
                      // InvalidOperationException

Rust rejects it at compile time, C# throws at runtime, Python and JS silently misbehave. The iterator is a reader; push needs the writer.

What does the compiler say?
Why does Rust reject the code above?

refused. push can reallocate the buffer and dangle the iterator's pointer — a use-after-free, prevented at compile time.

error[E0502]: cannot borrow `*items` as mutable because it is also borrowed as immutable
Lesson 09 · The memory model

Lifetimes

A reference must never outlive its data. In GC languages a reference keeps its target alive; Rust has no GC, so it proves the reference is safe. Returning a reference to a local shows the split.

Rust
fn f() -> &str {
    let s = make();
    &s                // error E0515: s dies here
}

function f(): string {
  const s = make();
  return s;           // fine — GC keeps s alive
}

def f() -> str:
    s = make()
    return s          # fine — object outlives the call

string F() {
    var s = Make();
    return s;         // fine — GC keeps s alive
}

You can't free memory and keep a pointer to it. In C that ships; in Rust it's E0515. A lifetime like &'a str is just the compiler naming how long a borrow may live.

What does the compiler say?
The signature promises a borrow of s, but the body returns a borrow of a local. Result?

refused. Fix: return an owned String, or borrow something that outlives the call.

error[E0515]: cannot return reference to local variable `s`
Lesson 10 · The memory model

Pointers & boxes

In GC languages every object is a hidden heap pointer you never choose. Rust stores values inline by default and makes you name the pointer: &T to borrow, Box<T> to own on the heap, Rc/Arc to share.

Rust
enum Tree {
    Leaf(i32),
    Node(Box<Tree>, Box<Tree>),  // Box = heap pointer
}                                 // (without it: infinite size)

type Tree =
  | { leaf: number }
  | { node: [Tree, Tree] };  // refs implicit; always heap

class Node:
    def __init__(self, left, right):
        self.left = left       # every attribute is a reference
        self.right = right

abstract record Tree;
record Node(Tree Left, Tree Right) : Tree;
// class/record fields are references (heap)

GC languages make every object a hidden heap pointer, so recursion just works. Rust stores values inline, so a recursive type needs an explicit Box — a single-owner heap pointer — to have a finite size.

The pointer family, plainest to most capable:

  • &T / &mut T — a borrow; owns nothing (Lesson 08).
  • Box<T> — a single-owner heap pointer, freed when it drops. For recursive types, trait objects (Box<dyn Trait>), and moving big values cheaply.
  • Rc<T> / Arc<T> — shared ownership by reference count (Lesson 13).
  • *const T / *mut T — raw pointers: no guarantees, only usable in unsafe.

agentj's run_turn is recursive — it re-enters itself through subagents. A recursive async fn would be an infinitely-sized state machine, so it carries #[async_recursion], which boxes the inner future: the same fix the quiz needs.

What does the compiler say?
Write the recursive type without the Box:
enum Tree {
    Leaf(i32),
    Node(Tree, Tree),
}

refused. A Node holds two Trees, each holding two Trees… with no fixed size. A Box is a fixed-size pointer to the heap, which breaks the recursion.

error[E0072]: recursive type `Tree` has infinite size help: insert some indirection (e.g., a `Box`, `Rc`, or `&`)
Lesson 11 · Abstraction

Traits

A trait is a named capability you implement for a type — nominal, not structural, and it can be added from outside the type's file. agentj converts a wire struct into TokenUsage via the standard From trait.

Rust
impl From<Wire> for Usage {
    fn from(w: Wire) -> Self { /* … */ }
}
let u: Usage = w.into();

// structural: any object of the right shape fits
function toUsage(w: Wire): Usage {
  return { tokens: w.tokens };
}

# duck typing: has the method? good enough
class Usage:
    @classmethod
    def from_wire(cls, w): ...

class Wire : IInto<Usage> {   // must declare it
    public Usage Into() => /* … */;
}

TS and Python match by shape (structural / duck typing). Rust and C# are nominal — but Rust can add the impl from outside the type's own file.

Check your model
You wrote impl From<Wire> for Usage. Does w.into() also work?

yes. A blanket impl in the standard library gives every From the mirror Into. One catch: .into() needs the target type known from context; Usage::from(w) never does.

Lesson 12 · Abstraction

Generics, closures, iterators

Generics compile a specialized copy per type (zero runtime cost). Closures capture by the ownership rules. Iterators are lazy — nothing runs until a consumer pulls, so a long chain is still one pass.

Rust
let out: Vec<_> = items.iter()
    .filter(|x| x.ok)
    .map(|x| x.id)
    .collect();       // one pass, runs at collect()

const out = items
  .filter(x => x.ok)  // builds an array
  .map(x => x.id);    // builds another array

out = [x.id for x in items if x.ok]
# comprehension: eager, one new list

var out = items
    .Where(x => x.Ok)   // LINQ: lazy
    .Select(x => x.Id); // runs when enumerated

Rust iterators and C# LINQ are lazy — no intermediate arrays. JS array methods and Python comprehensions build a new collection at each step.

What does the compiler say?
Warm a cache with a mapped side effect, never consumed:
items.iter().map(|x| warm_cache(x));

nothing happens. No consumer, no work.

warning: unused `Map` that must be used note: iterators are lazy and do nothing unless consumed

Use a for loop for side effects.

Lesson 13 · Concurrency

Arc, Mutex, atomics

One owner per value — but agentj's toolset is used by the UI, the turn, and subagents at once. Arc is shared ownership; Mutex lets many owners mutate safely.

Rust
let s = Arc::new(Mutex::new(map));
let s2 = s.clone();          // share across threads
s2.lock().unwrap().insert(k, v);

const s = new Map();
s.set(k, v);     // single-threaded: no locks,
                 // but no real parallelism either

lock = threading.Lock()
with lock:                   # GIL + explicit lock
    s[k] = v

lock (gate) {                // monitor on an object
    s[k] = v;
}

JS sidesteps this by being single-threaded; Python leans on the GIL. Rust shares across real threads and makes forgetting the lock a compile error — a data race is a type error.

What does the compiler say?
Drop the Mutex and mutate through the shared Arc<Tools> directly (&mut self). Result?

refused. &mut means exclusive; an Arc exists because access isn't. The Mutex is the minimal fix for "shared handle, occasional writes."

error[E0596]: cannot borrow data in an `Arc` as mutable
Lesson 14 · Concurrency

Async

agentj mostly waits — on the model, on child processes, on timers. An async fn is a state machine that parks at each .await. Futures are lazy: nothing runs until awaited or spawned.

Rust
let fut = fetch();   // nothing runs yet
let out = fut.await; // now it runs

const p = fetch();   // ALREADY running
const out = await p; // just waits for it

coro = fetch()       # not running yet
out = await coro     # runs under the event loop

var t = FetchAsync(); // already running (hot task)
var out = await t;    // waits for it

Futures in Rust (and coroutines in Python) are lazy. JS promises and C# tasks are hot — they start the moment you create them. Rust also has no ambient event loop: you pick a runtime (tokio) and spawn explicitly.

Not everything is a compile error
agentj drains a child process's stdout in a spawned task while waiting. Skip that — just wait() then read. Result?

compiles, hangs. A chatty child fills the pipe buffer and blocks; the parent blocks in wait(); deadlock. Rust guarantees memory and race safety — deadlocks and logic bugs are still yours.

Lesson 15 · Craft

Tests & modules

Rust's test runner is built in. Tests live beside the code under #[cfg(test)] — compiled only for cargo test, absent from the shipped binary.

Rust
#[cfg(test)]        // same file; excluded from release
mod tests {
    #[test]
    fn it_works() { assert_eq!(2 + 2, 4); }
}

// sum.test.ts — a separate file
import { sum } from "./sum";
test("adds", () => { expect(sum(2, 2)).toBe(4); });

# test_sum.py — a separate file
def test_adds():
    assert sum(2, 2) == 4

// SumTests.cs — usually a separate project
[Fact]
public void Adds() => Assert.Equal(4, Sum(2, 2));

The others bolt on an external framework and a separate file. Rust ships the runner and keeps tests next to the code.

Modules mirror the file tree: a parent declares each child with mod files;, and pub(super) / pub(crate) / pub control who can reach in. That's how safe_resolve stays callable only inside src/tools/.
