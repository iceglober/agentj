Current state vs. the simplified state (now implemented)

The request: reset the prompts, strip every mention of SPEAR and strict protocols, remove all steering/supervisor usage, and make the prime prompt very simple. That is done. Toggle to compare what the system was against what it now is.

The prime prompt was a wall of doctrine

A multi-page instructions() built the whole SPEAR loop (Scope · Plan · Execute · Assess · Resolve) as a literal spine, plus a separate epic_doctrine() block, the design-contest doctrine, the meta-plan/frontier ceremony, and a dozen "judicious re-adds." It read as an accreted pile — and it still default-dove a greenfield build instead of exploring.

A supervisor steered every turn ≈600 lines

  • spear_nudge — re-anchor after 12 direct calls
  • step_budget_nudge — "8 steps remain, converge now"
  • greenfield_edit_gate — block the first scaffold write into an empty repo
  • ASSESS / RESOLVE-ship / RESOLVE-completeness / epic-merge finishing gates
  • phase() inference → a live SPEAR-spine panel in the TUI

A continuation judge second-guessed every stop

On idle or step-budget exhaustion, a fresh model context was asked CONTINUE/STOP and could re-arm the budget — src/agent/continuation.rs, up to 6 extensions per turn.

Steering was a first-class transcript citizen

LineKind::Steering, a "Show steering" Ctrl-P toggle, tagged [supervisor: …] notes. The Ctrl-P menu carried 9 items.

The prime prompt is four tagged sections

This is the entire instructions() body now — four ideas, each its own opening/closing tag, nothing else about how to work:
<explore> Always explore and always plan before you take any action. Build a HIGH-CONFIDENCE understanding of the context from HARD EVIDENCE — read the code, run commands, look at the real files — before you act; never work from assumption. </explore> <subagents> Use subagents as much as possible, and launch them in PARALLEL as much as possible — one `run_subagents` call with several independent tasks. This applies to EVERY kind of task, not just building things: answering a question about the code, tracking down a bug, reviewing a change, refactoring, or auditing all fan out the same way. Delegate the reading and exploring to them; you synthesize and decide. </subagents> <align> For anything beyond a trivial change, ALIGN with the user before you build. When a picture would land faster than prose — an interface, a data model, a page layout, a multi-step flow — save a `blueprint` (`save_artifact` with format:"html"): a self-contained interactive page using mockups, hierarchy, and diagrams that shows what you understand and what you propose. It opens in their browser. Track the work itself in a `todos` artifact (pending / in-progress / done, with evidence) and keep it current; hold the settled approach in `plan`. </align> <verify> Always verify the user's request has been fulfilled beyond a shadow of a doubt. Check your work no matter what the task was. If you answered a question about the codebase, double-check your reasoning and make sure your conclusion rests on hard evidence. If you made a code change, run the tests, run the code, and manually exercise it — cURL an endpoint, run a script, or drive the UI with a browser. </verify>

The loop is just: call → run tools → repeat

// src/agent/mod.rs — run_turn, stripped to the skeleton loop { if step >= step_budget { emit StepLimit; return; } // hard stop, no judge drain background-job nudges; // primary loop only if prompt too large && already mutated { compact old tool results; } turn = llm.chat(messages, tool_specs); emit Thinking / Message; if no tool_calls { wait on running jobs, else Done; } for call in tool_calls { run it (run_subagents is intercepted); } }

What survived — and why it isn't steering

  • Typed subagents — scout/planner/reviewer/executor (oracle folded away), each a scoped toolset. Identities are plain role descriptions (no SPEAR); the reviewer carries a MECE rubric.
  • Frontier resume — moved out of the supervisor to a plain frontier_resume() in agent/mod.rs; a --resume/--continue convenience that re-injects your plan artifact, not a nudge.
  • Context compaction — still elides old tool bodies once the turn has mutated a file; now gated on a local mutated flag instead of the supervisor.
  • Background jobs, MCP, sessions — untouched. (The epic / engineer / DAG / advanced-delegation feature is gone entirely — see the note below.)
  • Ctrl-P menu is now 8 items; LineKind is Plain/User/Assistant/Tool/Note/Thinking.
Deleted outright: supervisor.rs, continuation.rs, view/spear.rs, the SpearPhase type, the AgentEvent::Phase variant, LineKind::Steering + its toggle, epic_doctrine() — and, in the KISS pass: the web_check tool, the engineer_start tool and the entire epic / engineer / DAG / advanced-delegation feature (engineers.rs, tui/epic.rs, view/dag.rs, view/epic.rs, the manage/epic_automerge config, the two Ctrl-P toggles, the /epic command). The oracle subagent type was folded away and delegate was renamed run_subagents.

Agents & their prompts

There is one primary agent (PRIME) and four subagent types dispatched by the run_subagents tool. Each agent's system prompt is built from opening/closing tagged sections: PRIME gets <identity> <context> <explore> <subagents> <align> <verify> (+ <project_docs> <skills> when present); a subagent gets <role> <contract> (+ <rubric> for the reviewer) then the same context/docs/skills. Every prompt string below is verbatim from the source.

Shared subagent contract (prepended-to by every type identity)
Do EXACTLY the one sub-task you're given — nothing more, no scope creep. Your efficiency bar is strict: the FEWEST tool calls that produce hard evidence. Batch related commands into one call; run ONE broad query and filter its output rather than repeating a query with shifted parameters; read only the files your brief names or your evidence demands. (Exception: reproducing nondeterministic behavior — flaky tests, races — is done by rerunning the SAME command; that repetition is the method, not waste.) If you discovered independent follow-up work outside your sub-task, do NOT do it — end with a `frontier:` line listing it so the main agent can schedule it. Your entire final reply becomes the result handed back to the main agent — tight and self-contained, no filler, no meta-commentary.

Tool catalogue

All 14 built-in tools plus the MCP meta-tool, with signatures and gating. core = always available to the primary loop; subagents see only what their type allows. primary / session / mcp mark the conditionally-advertised tools.

ToolSignatureGateWhat it does

Per-type tool scoping

Enforced twice: tool_specs(…, agent_type) filters what's advertised, and Tools::call refuses a disallowed call at dispatch. The type allowlist constant READ_ONLY = read_file, list_dir, glob, grep, mcp_find_tools, job_check, read_artifact — but subagents run without a session store (scoped_to sets session: None), so the artifact tools are never actually advertised to them; the rows below show what a subagent truly sees.

TypeAccessAllowed built-ins

Typed artifacts

Artifacts persist in the session store (outside the repo, keyed to the session). save_artifact takes a format — markdown (default) or html. An html save is written as <name>.html and opened in the user's browser, which is the alignment moment. Three are conventional:

ArtifactFormatRole · lifecycle
planmarkdownThe settled approach + rationale. Written once the design is decided, revised only on new info. Re-injected on resume.
todosmarkdownThe execution checklist — pending / in-progress / done, with evidence. Churns every step. Re-injected on resume, and led with (it's what tells a resumed run what's left).
blueprinthtmlAn interactive visual alignment page — mockups, hierarchy, diagrams. Opens in the user's browser on save. For the user, not the model context; not re-injected.
